# COS3 SEF 文件字节序分析

## 关键发现

根据 **COS3 规范第 413 行**的明确规定：

> **链接文件和可加载文件中 i32、s16、s32、u16、u32 类型的数据项应按照小端顺序存储。**

这意味着：
- ✅ **SEF 文件使用小端字节序（Little-Endian）**
- ✅ 所有多字节整数（u16, u32）都是低位字节在前

---

## 附录 F.1 示例数据分析

### 原始十六进制数据

```
66657300 00000001
```

### 字节序解析

#### 1. sef_type (u32, 小端)

**内存中的字节序列**：`66 65 73 00`

按照**小端顺序**解析：
```
地址:    [0]    [1]    [2]    [3]
字节:    0x66   0x65   0x73   0x00
权重:   LSB(最低)              MSB(最高)

数值 = 0x00 * 2^24 + 0x73 * 2^16 + 0x65 * 2^8 + 0x66 * 2^0
     = 0x00736566
```

**转换为 ASCII**：
- `0x66` = 'f'
- `0x65` = 'e'
- `0x73` = 's'
- `0x00` = '\0'

字符串表示：**"fes\0"** （注意：这是小端存储的结果！）

**但规范要求的是 "sef\0"**，让我们重新检查...

实际上，如果我们把 `0x00736566` 看作一个 u32 值：
- 在源代码中写作：`'00736566'`（规范表 10）
- 在内存中小端存储为：`66 65 73 00`
- 读取时按小端重组为：`0x00736566`
- 解释为字符串时：**从低地址到高地址读** → `"fes\0"` ❌

**这里有个误解！** 让我们重新理解规范的意图。

---

## 正确的理解方式

### 规范表 10 的含义

规范表 10 说：
```
可加载文件 | '00736566' | GB/T 1988编码字符"sef"
```

这里的 `'00736566'` 是一个 **u32 常量的十六进制表示**，不是字符串字面量！

**正确的解释**：

1. **逻辑值**：`sef_type = 0x00736566`
2. **小端存储**：在文件中存储为字节序列 `66 65 73 00`
3. **ASCII 对应**：
   - `0x73` = 's'
   - `0x65` = 'e'
   - `0x66` = 'f'
   - `0x00` = '\0'

所以 `0x00736566` 这个数值对应的 ASCII 字符串是 **"sef\0"**（从高位到低位读）。

### 验证计算

```c
// 如果我们将 0x00736566 分解为字节（大端视角，即人类可读的十六进制）
uint32_t value = 0x00736566;

// 在大端系统中，内存布局是：
// [0x00] [0x73] [0x65] [0x66]
//  \0     's'    'e'    'f'

// 在小端系统中，内存布局是：
// [0x66] [0x65] [0x73] [0x00]
//  'f'    'e'    's'    \0

// 但当我们用 u32 读取时，无论系统字节序如何，都会得到相同的数值 0x00736566
```

**关键点**：
- 规范定义的是**数值** `0x00736566`
- 文件中使用**小端存储**这个数值
- 当程序以小端方式读取这 4 个字节时，会得到数值 `0x00736566`
- 这个数值的 ASCII 解释是 "sef\0"（从最高字节到最低字节）

## 附录 F.1 数据的完整解析

### 文件头（8 字节）

```
偏移 0-3: 66 65 73 00  → sef_type = 0x00736566 (小端)
偏移 4-7: 00 00 00 01  → version = 0x00000001 (小端) = 版本 1.0.0.0
```

**验证**：
```c
// 读取 sef_type（小端）
uint8_t bytes[4] = {0x66, 0x65, 0x73, 0x00};
uint32_t sef_type = bytes[0] | (bytes[1] << 8) | (bytes[2] << 16) | (bytes[3] << 24);
// sef_type = 0x66 | 0x6500 | 0x730000 | 0x00000000
// sef_type = 0x00736566 ✓

// 读取 version（小端）
uint8_t ver_bytes[4] = {0x00, 0x00, 0x00, 0x01};
uint32_t version = ver_bytes[0] | (ver_bytes[1] << 8) | (ver_bytes[2] << 16) | (ver_bytes[3] << 24);
// version = 0x00 | 0x0000 | 0x000000 | 0x01000000
// version = 0x01000000 ✗ 等等，这不对！
```

**发现问题了！** 让我重新检查...

实际上，如果 version 是 `00 00 00 01`，按小端读取应该是：
```
version = 0x00 + (0x00 << 8) + (0x00 << 16) + (0x01 << 24)
        = 0x01000000
```

但这表示版本 `1.0.0.0`（major=1），而不是 `0.0.0.1`。

## 结论

### ✅ 确认的事实

1. **COS3 SEF 文件使用小端字节序**（规范第 413 行明确规定）

2. **魔术字 `0x00736566` 的小端存储**：
   - 数值：`0x00736566`
   - 小端字节序列：`66 65 73 00`
   - ASCII 解释：从高位到低位是 "sef\0"

3. **附录 F.1 的文件头**：
   ```
   66 65 73 00  → sef_type = 0x00736566 ("sef\0") ✓
   00 00 00 01  → version = 0x01000000 (版本 1.0.0.0)
   ```

```c
// 正确的小端读取方式
uint32_t read_u32_le(const uint8_t *data) {
    return (uint32_t)data[0] |
           ((uint32_t)data[1] << 8) |
           ((uint32_t)data[2] << 16) |
           ((uint32_t)data[3] << 24);
}

// 使用时
uint32_t sef_type = read_u32_le(sef_data + 0);
uint32_t version = read_u32_le(sef_data + 4);

    // 错误处理
}
```

### 📋 修复建议

1. **修改魔术字定义**：
   ```c
   // gcos_vm/src/gcos_loader.c
   #define SEF_MAGIC           0x00736566  /* "sef\0" */

   ```

2. **添加小端读取辅助函数**：
   ```c

   }
   
   static inline uint32_t read_u32_le(const uint8_t *data) {
       return (uint32_t)data[0] |
              ((uint32_t)data[1] << 8) |
              ((uint32_t)data[2] << 16) |
              ((uint32_t)data[3] << 24);
   }
   ```

3. **更新所有多字节字段读取**：
   - 段大小（u32）
   - 版本号（u32）
   - 各种计数和索引（u16/u32）


4. **版本号格式（附录 B）**：
   ```
   u32 version 结构：
   - Byte 3 (MSB): 主版本号 (major)
   - Byte 2:       次版本号 (minor)
   - Byte 1:       修订号 (revision)
   - Byte 0 (LSB): 内部版本号 (internal)
   
   示例：0x01000000 = v1.0.0.0
         0x02010305 = v2.1.3.5
   ```

### ⚠️ 当前 GCOS 实现的问题

**问题 1：魔术字定义错误**

当前实现直接映射结构体，没有显式的小端转换，不符合规范要求。
