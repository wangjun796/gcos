## 🔍 与 cref 和 WAMR 的深度对比

### 架构对比

| 特性 | GCOS | cref (JavaCard) | WAMR (WebAssembly) |
|------|------|----------------|-------------------|
| **执行模型** | 栈式字节码 | 栈式字节码 | 栈式 + 寄存器混合 |
| **内存模型** | 静态分配为主 | 动态分配 + GC | 线性内存 + GC |
| **模块化** | 模块 + 应用实例 | 包 (Package) + Applet | Module + Instance |
| **链接机制** | GRT (32-bit 紧凑) | Constant Pool | Import/Export |
| **安全性** | 基础隔离 | Firewall + GP Security | Sandbox + WASI |
| **持久化** | eflash FTL | EEPROM 模拟 | 无（浏览器环境） |
| **工具链** | 缺失 | javac + capgen | wat2wasm + wamrc |
| **适用场景** | 智能卡/嵌入式 | 智能卡/Javacard | IoT/Edge Computing |

### 关键差异分析

#### 1. 内存管理策略

**GCOS**:
- ✅ 零动态内存分配（生产环境）
- ✅ 1-based 堆地址语义（避免 0 歧义）
- ⚠️ 静态分配限制了灵活性

**cref**:
- ⚠️ 使用动态内存分配
- ⚠️ 需要垃圾回收（GC）
- ✅ 更灵活的内存使用

**WAMR**:
- ✅ 线性内存模型，简单高效
- ⚠️ 可选 GC（取决于配置）
- ✅ 支持内存增长

**结论**: GCOS 的静态分配适合资源受限的智能卡，但牺牲了灵活性。

#### 2. 模块链接机制

**GCOS - GRT (全局引用表)**:
```c
// 32-bit 紧凑格式
typedef struct {
    u8  module_id;      // 8-bit 模块 ID
    u24 address;        // 24-bit 地址（偏移量）
} GCOSGlobalRefEntry;
```
- ✅ 节省空间（每个条目仅 4 字节）
- ⚠️ 限制：最多 256 个模块，地址范围 16MB
- ✅ 批量回收机制高效
