# COS3 规范 vs WebAssembly 核心差异分析

## 📋 文档信息

- **版本**: v1.0
- **日期**: 2026-05-09
- **作者**: GCOS VM Development Team
- **参考标准**: 
  - GB/T 44901.3-XXXX (COS3规范)
  - W3C WebAssembly Core Specification (2019)

## 1. 概述

本文档详细分析 **COS3规范**（中国国家标准，智能卡虚拟机）与 **WebAssembly**（W3C国际标准，通用虚拟机）的核心差异，为GCOS VM的实现提供技术参考。

### 1.1 定位对比

| 维度 | COS3 | WebAssembly |
|------|------|-------------|
| **标准性质** | 中国国家标准 (GB/T) | W3C国际标准 |
| **应用场景** | 智能卡、安全芯片、身份识别设备 | Web浏览器、服务器、IoT设备 |
| **目标平台** | 资源受限嵌入式系统 (RAM < 32KB) | 通用计算平台 (RAM无限制) |
| **设计理念** | 安全性优先、事务支持、应用隔离 | 性能优先、跨平台、互操作性 |
| **编程语言** | C语言子集 → SEF字节码 | C/C++/Rust等 → WASM字节码 |
| **文件格式** | .sef (可加载文件) | .wasm (WebAssembly模块) |

```
│  │  • 删除管理                      │  │
│  └──────────────────────────────────┘  │
│  ┌──────────────────────────────────┐  │
│  │  执行器 (Executor)               │  │
│  │  • 字节码解释器                  │  │

│  │  • 应用执行管理                  │  │
│  └──────────────────────────────────┘  │
│  ┌──────────────────────────────────┐  │
│  │  安全管理 (Security)             │  │

               │
```

```
└─────────────────────────────────────────┘
```

### 2.2 核心差异总结

| 特性 | COS3 | WebAssembly | 影响 |
|------|------|-------------|------|
| **应用安装器** | ✅ 内置 (加载/安装/删除) | ❌ 由宿主环境提供 | COS3需要实现完整的生命周期管理 |
| **事务管理** | ✅ 内置 (原子性保证) | ❌ 不支持 | COS3需要实现备份/恢复机制 |
| **多通道支持** | ✅ 8个逻辑通道 | ❌ 单实例 | COS3需要通道隔离机制 |
| **应用隔离** | ✅ 强制 (运行域隔离) | ⚠️ 沙箱 (模块级) | COS3需要更细粒度的访问控制 |
| **持久化存储** | ✅ 堆数据掉电不丢失 | ❌ 内存易失 | COS3需要Flash集成 |
| **垃圾回收** | ❌ 手动管理 | ❌ 手动管理 (目前) | 两者相似 |
| **浮点支持** | ❌ 不支持float/double | ✅ 支持f32/f64 | COS3仅支持整数运算 |
| **64位整数** | ❌ 不支持i64 | ✅ 支持i64 | COS3最大32位 |
| **引用类型** | ❌ 不支持 | ✅ funcref/externref | COS3使用指针替代 |

## 3. 内存模型差异 ⭐⭐⭐

### 3.1 COS3 内存模型

COS3采用**分区内存模型**，区分易失性和非易失性存储：

```
运行时数据区 (Runtime Data Area)
┌─────────────────────────────────────────┐
│  模块程序区 (Module Code)                │
│  • 非易失性 (Non-volatile)              │
│  • 存放字节码指令                        │
│  • 大小: 16 KB                          │
├─────────────────────────────────────────┤
│  堆 (Heap)                               │
│  • 非易失性 (Non-volatile) ⭐           │
│  • 掉电数据不丢失                        │
│  • 存放:                                 │
│    - 应用域数据 (App Domain Data)       │
│    - 引用域数据 (Ref Domain Data)       │
│    - 持久性数据 (Persistent Data)       │
│    - 模块域数据 (Module Domain Data)    │
│    - 临时动态数据 (Temp Dynamic Data)   │
│  • 大小: 8 KB                           │
├─────────────────────────────────────────┤
│  全局数据区 (Global Data)                │
│  • 易失性 (Volatile)                    │
│  • 上电复位后丢失                        │
│  • 存放:                                 │
│    - 模块全局数据 (Module Global)       │
│    - 临时静态数据 (Temp Static)         │
│    - 临时动态数据副本                   │
│  • 大小: 4 KB                           │
├─────────────────────────────────────────┤
│  间接访问变量栈 (Indirect Stack)         │
│  • 易失性 (Volatile)                    │
│  • 存放组合数据类型元素                  │
│  • 栈单元: 16 字节                      │
│  • 大小: 64 × 16B = 1 KB                │
├─────────────────────────────────────────┤
│  执行器栈 (Executor Stack)               │
│  • 易失性 (Volatile)                    │
│  • 存放栈帧、参数、局部变量、中间结果    │
│  • 栈单元: 4 字节                       │
│  • 大小: 256 × 4B = 1 KB                │
└─────────────────────────────────────────┘
  ↑ 低地址                              高地址 ↑
```

#### COS3 数据类型与存储位置

| 数据类型 | 存储位置 | 易失性 | 作用域 | 生命周期 |
|----------|----------|--------|--------|----------|
| **模块全局数据** | 全局数据区 | ✅ 易失 | 模块范围 | 应用取消选择后丢失 |
| **模块只读数据** | ROM | ❌ 非易失 | 模块范围 | 永久，不可修改 |
| **模块局部数据** | 执行器栈 | ✅ 易失 | 函数范围 | 函数退出后丢失 |
| **模块域数据** | 堆 | ❌ 非易失 | 模块范围 | 永久，模块删除时回收 |
| **临时静态数据** | 全局数据区 + 堆头 | ✅ 易失 | 应用实例 | 上电复位后丢失 |
| **临时动态数据** | 全局数据区 + 堆头 | ✅ 易失 | 应用实例 | 应用取消选择后丢失 |
| **跨域数据** | 全局数据区 + 堆头 | ✅ 易失 | 所有应用 | 上电复位后丢失 |
| **持久性数据** | 堆 | ❌ 非易失 | 应用实例 | 永久，受事务保护 |
| **应用域数据** | 堆 | ❌ 非易失 | 应用实例 | 永久，受事务保护 |
| **引用域数据** | 堆 | ❌ 非易失 | 应用实例 | 永久，受事务保护 |

### 3.2 WebAssembly 内存模型

WebAssembly采用**线性内存模型**：

```
Linear Memory (连续字节数组)
┌─────────────────────────────────────────┐
│  Memory Page 0 (64 KB)                  │
│  • 初始内存                             │
│  • 可按页增长 (每页64KB)                │
│  • 最大可达 4GB (32位) 或 16EB (64位)  │
├─────────────────────────────────────────┤
│  Memory Page 1 (64 KB)                  │
│  • 动态增长                             │
├─────────────────────────────────────────┤
│  Memory Page N (64 KB)                  │
│  • ...                                  │
└─────────────────────────────────────────┘

特点:
• 单一连续地址空间
• 字节寻址 (byte-addressable)
• 沙箱化 (不能访问模块外内存)
• 易失性 (重启后丢失)
• 可与JavaScript共享
```

### 3.3 关键差异对比

| 特性 | COS3 | WebAssembly |
|------|------|-------------|
| **内存布局** | 分区固定大小 (栈/堆/全局数据分离) | 单一连续线性内存 |
| **持久化** | ✅ 堆数据非易失 (掉电保持) | ❌ 全部易失 |
| **内存增长** | ❌ 固定大小 (编译时确定) | ✅ 可按页动态增长 |
| **地址空间** | 多个独立区域 (栈/堆/全局) | 单一扁平地址空间 |
| **内存保护** | ✅ 分区隔离 + 边界检查 | ✅ 沙箱边界检查 |
| **最大内存** | ~32 KB (典型智能卡) | 4 GB (32位) / 16 EB (64位) |
| **内存类型** | 易失性 + 非易失性混合 | 仅易失性 |
| **共享内存** | ❌ 应用隔离 | ✅ 可通过SharedArrayBuffer共享 |

### 3.4 对实现的影响

**COS3 VM 需要**:
1. ✅ 实现多种存储区域的管理器
2. ✅ 集成Flash存储后端 (eflash)
3. ✅ 实现掉电恢复机制
4. ✅ 实现内存分区分配策略
5. ✅ 实现事务备份/恢复 (针对非易失数据)

**WebAssembly VM 只需**:
1. ✅ 实现单一内存管理器
2. ✅ 支持内存动态增长
3. ✅ 实现边界检查

## 4. 指令集差异

### 4.1 数据类型支持

| 类型 | COS3 | WebAssembly | 说明 |
|------|------|-------------|------|
| **8位有符号** | ✅ `signed char` | ✅ `i32` (模拟) | COS3原生支持 |
| **8位无符号** | ✅ `unsigned char` | ✅ `i32` (模拟) | COS3原生支持 |
| **16位有符号** | ✅ `signed short` | ✅ `i32` (模拟) | COS3原生支持 |
| **16位无符号** | ✅ `unsigned short` | ✅ `i32` (模拟) | COS3原生支持 |
| **32位有符号** | ✅ `signed int` | ✅ `i32` | 两者都支持 |
| **32位无符号** | ✅ `unsigned int` | ✅ `i32` (模拟) | COS3原生支持 |
| **64位整数** | ❌ 不支持 | ✅ `i64` | WASM特有 |
| **32位浮点** | ❌ 不支持 | ✅ `f32` | WASM特有 |
| **64位浮点** | ❌ 不支持 | ✅ `f64` | WASM特有 |
| **引用类型** | ❌ 不支持 | ✅ `funcref`, `externref` | WASM 2.0+ |

**结论**: COS3仅支持整数运算 (8/16/32位)，不支持浮点和64位整数。

### 4.2 指令分类对比

#### COS3 指令分类 (表48)

1. **控制指令**: trap, nop, br, beqz, bnez, call, ret
2. **数值指令**: 
   - 常量: ldc.0~8, ldc.i32, ldc.m1
   - 算术: add, sub, mul, divs/divu, mods/modu
   - 位运算: and, or, xor, shl, shrs, shru, rotl, rotr
   - 比较: eqz, eq, ne, lts/l tu, gts/gtu, les/leu, ges/geu
3. **变量指令**: ldt.0~8, ldt, stt.0~8, stt
4. **内存指令**: lda, sta, ldind, stind
5. **异常处理指令**: throw, try, catch, endtry
6. **复合指令**: dup, swap, over, rot

#### WebAssembly 指令分类

1. **控制指令**: unreachable, nop, block, loop, if, else, end, br, br_if, br_table, return, call, call_indirect
2. **参数指令**: local.get, local.set, local.tee, global.get, global.set
3. **内存指令**: memory.size, memory.grow, load, store
4. **数值指令**:
   - 常量: i32.const, i64.const, f32.const, f64.const
   - 算术: i32.add, i32.sub, i32.mul, i32.div_s/u, i32.rem_s/u
   - 位运算: i32.and, i32.or, i32.xor, i32.shl, i32.shr_s/u
   - 比较: i32.eqz, i32.eq, i32.ne, i32.lt_s/u, ...
   - 转换: i32.wrap_i64, i64.extend_i32_s/u, ...
5. **表操作**: table.get, table.set, table.size, table.grow

### 4.3 关键指令差异

| 功能 | COS3 | WebAssembly | 说明 |
|------|------|-------------|------|
| **陷阱** | `trap` (0x00) | `unreachable` | 类似 |
| **无条件跳转** | `br.s8/s16` | `br` | COS3支持8/16位偏移 |
| **条件跳转** | `beqz.s8/s16`, `bnez.s8/s16` | `br_if` | COS3更具体 |
| **函数调用** | `call.u16`, `callin.u8`, `callex.u8` | `call`, `call_indirect` | COS3区分内部/外部 |
| **返回值** | `iret` (有返回), `ret` (无返回) | `return` (统一) | COS3区分更细 |
| **常量加载** | `ldc.0~8`, `ldc.i32`, `ldc.m1` | `i32.const` | COS3优化小常量 |
| **局部变量** | `ldt.0~8`, `ldt`, `stt.0~8`, `stt` | `local.get/set/tee` | COS3优化小索引 |
| **除零检查** | ✅ 抛出异常 | ✅ 未定义行为 | COS3更严格 |
| **溢出检查** | ✅ `add.ovf`, `sub.ovf`, `mul.ovf` | ❌ 截断 | COS3更安全 |
| **异常处理** | ✅ `throw`, `try`, `catch`, `endtry` | ❌ 不支持 (提案中) | COS3内置支持 |
| **内存增长** | ❌ 不支持 | ✅ `memory.grow` | COS3固定大小 |

### 4.4 操作码编码差异

#### COS3 操作码

- **单字节**: `0x00` ~ `0xFB` (252条)
- **双字节**: `0xFC` ~ `0xFE` 前缀 + 第二字节 (可扩展)
- **保留**: `0xFF`

#### WebAssembly 操作码

- **单字节**: `0x00` ~ `0xFF`
- **多字节**: 使用LEB128编码的操作数
- **无前缀**: 直接编码

**结论**: COS3采用双字节扩展机制，WASM使用LEB128可变长度编码。

## 5. 应用生命周期管理差异 ⭐⭐⭐

### 5.1 COS3 应用生命周期

COS3定义了完整的应用生命周期管理：

```
应用生命周期状态机:

[已安装] ←→ [可选择] ←→ [已选择/运行中]
    ↓           ↓              ↓
[已锁定]    [已个性化]    [已终止]
```

#### 状态转换

| 状态 | 描述 | 触发事件 |
|------|------|----------|
| **已安装** (INSTALLED) | 应用已安装但未激活 | 安装完成 |
| **可选择** (SELECTABLE) | 应用可被选择 | 个性化完成 |
| **已选择** (SELECTED) | 应用正在运行 | SELECT命令 |
| **已个性化** (PERSONALIZED) | 应用已配置密钥等 | 个性化命令 |
| **已锁定** (LOCKED) | 应用被锁定 | 锁定命令 |
| **已终止** (TERMINATED) | 应用被卸载 | 删除命令 |

#### 关键机制

1. **应用选择 (SELECT)**
   - 查找匹配AID的应用
   - 检查生命周期状态
   - 调用应用选择接口
   - 分配模块全局数据
   - 分配临时动态数据

2. **应用取消选择 (DESELECT)**
   - 调用应用取消选择接口
   - 回收模块全局数据
   - 回收临时动态数据
   - 清除通道记录

3. **多通道支持**
   - 最多8个逻辑通道
   - 每个通道独立选择应用
   - 同一应用可在多通道运行
   - 通道间数据隔离

### 5.2 WebAssembly 应用生命周期

WebAssembly**没有**内置的应用生命周期管理：

```
WASM 模块生命周期:

[加载] → [实例化] → [执行] → [销毁]
   ↓          ↓          ↓        ↓
 Parse    Instantiate  Execute  Garbage Collect
```

- **加载**: 由宿主环境负责 (.wasm文件读取)
- **实例化**: 创建Module实例，链接导入/导出
- **执行**: 调用导出函数
- **销毁**: 由垃圾回收器管理

**关键区别**:
- ❌ 无应用选择/取消选择机制
- ❌ 无多通道支持
- ❌ 无生命周期状态管理
- ❌ 无AID概念
- ✅ 由宿主环境 (如浏览器) 管理

### 5.3 对实现的影响

**COS3 VM 必须实现**:
1. ✅ 应用注册表 (AID → 应用实例映射)
2. ✅ 生命周期状态机
3. ✅ SELECT/DESELECT命令处理
4. ✅ 多通道管理器 (8个通道)
5. ✅ 通道数据隔离
6. ✅ 应用查找算法

**WebAssembly VM 不需要**:
- 这些功能由宿主环境提供

## 6. 事务管理差异 ⭐⭐⭐

### 6.1 COS3 事务管理

COS3**内置**事务管理机制，确保数据原子性：

```c
// COS3 事务API
TRY {
    // 开始事务
    transaction_begin();
    
    // 更新持久性数据
    update_persistent_data(...);
    update_app_domain_data(...);
    update_ref_domain_data(...);
    
    // 提交事务
    transaction_commit();
}
CATCH {
    // 异常时自动回滚
    transaction_abort();
}
```

#### 事务特性

| 特性 | 说明 |
|------|------|
| **原子性** | 要么全部成功，要么全部回滚 |
| **备份机制** | 事务开始时备份堆数据 |
| **回滚机制** | 事务中止时恢复备份数据 |
| **嵌套支持** | 支持嵌套事务 (有限深度) |
| **保护范围** | 持久性数据、应用域数据、引用域数据 |
| **自动取消** | 应用选择/取消选择时取消未结束事务 |

#### 事务流程

```
transaction_begin():
  1. 备份堆数据到临时缓冲区
  2. 备份全局数据区 (可选)
  3. 设置 active = true
  4. nesting_level++

transaction_commit():
  1. 释放备份数据
  2. active = false
  3. nesting_level--

transaction_abort():
  1. 从备份恢复堆数据
  2. 从备份恢复全局数据区
  3. 释放备份数据
  4. active = false
  5. nesting_level--
```

### 6.2 WebAssembly 事务管理

WebAssembly**不支持**事务：

- ❌ 无内置事务机制
- ❌ 无原子性保证
- ❌ 无备份/恢复机制
- ✅ 由应用程序自行实现 (如果需要)

### 6.3 对实现的影响

**COS3 VM 必须实现**:
1. ✅ 事务上下文管理
2. ✅ 数据备份机制 (堆 + 全局数据)
3. ✅ 数据恢复机制
4. ✅ 嵌套事务支持
5. ✅ 与Flash存储集成 (持久化)
6. ✅ TRY-CATCH异常处理集成

**WebAssembly VM 不需要**:
- 事务是应用层责任

## 7. 安全管理差异 ⭐⭐

### 7.1 COS3 安全管理

COS3提供**多层次**安全管理：

#### 7.1.1 运行时安全管理

1. **应用隔离**
   - 运行域隔离 (每个应用独立数据空间)
   - 模块数据隔离
   - 应用数据隔离
   - 模块程序隔离

2. **接口授权**
   - 基于授权表的接口调用
   - 只有被授权应用才能调用其他应用接口
   - 防止越权访问

3. **指令执行安全**
   - 边界检查 (数组、内存访问)
   - 类型检查
   - 栈溢出保护
   - 除零检查

#### 7.1.2 系统安全管理

- 密码运算 (AES, RSA, ECC)
- 密钥管理
- 权限管理
- 安全启动

### 7.2 WebAssembly 安全管理

WebAssembly提供**沙箱**安全模型：

1. **内存沙箱**
   - 不能访问模块外内存
   - 边界检查 (load/store指令)
   - 不能执行任意代码

2. **类型安全**
   - 静态类型检查 (加载时验证)
   - 运行时类型检查

3. **控制流完整性**
   - 结构化控制流 (block/loop/if)
   - 不能跳转到任意地址

4. **宿主隔离**
   - 不能直接访问系统资源
   - 必须通过导入函数

### 7.3 关键差异对比

| 安全特性 | COS3 | WebAssembly |
|----------|------|-------------|
| **应用隔离** | ✅ 强制 (运行域级别) | ⚠️ 模块级沙箱 |
| **接口授权** | ✅ 基于授权表 | ❌ 由宿主控制 |
| **内存隔离** | ✅ 分区隔离 + 边界检查 | ✅ 沙箱边界检查 |
| **类型安全** | ✅ 运行时检查 | ✅ 加载时验证 + 运行时 |
| **栈保护** | ✅ 溢出检测 | ✅ 溢出检测 |
| **除零检查** | ✅ 抛出异常 | ⚠️ 未定义行为 |
| **溢出检查** | ✅ 可选 (ovf指令) | ❌ 截断 |
| **事务保护** | ✅ 内置 | ❌ 应用层实现 |
| **多租户** | ✅ 8通道隔离 | ❌ 单实例 |

### 7.4 对实现的影响

**COS3 VM 需要更强的安全检查**:
1. ✅ 运行域隔离机制
2. ✅ 接口授权表管理
3. ✅ 严格的边界检查
4. ✅ 事务一致性检查
5. ✅ 多通道数据隔离

## 8. 文件格式差异

### 8.1 COS3 SEF 文件格式

SEF (Secure Executable Format) 可加载文件结构：

```
SEF File Header (12 bytes):
┌─────────────────────────────────┐
│ sef_type (u32): 0x00736566      │  // "sef"
│ version (u32)                   │
│ section_count (u32)             │
└─────────────────────────────────┘

Sections (按顺序):
┌─────────────────────────────────┐
│ 1. First Section (0x01, 必选)   │  // 模块信息、导入信息
│ 2. Import Section (0x02, 可选)  │  // 导入函数
│ 3. Function Section (0x03, 必选)│  // 函数空间信息
│ 4. App Section (0x04, 可选)     │  // 应用安装信息
│ 5. Global Section (0x05, 必选)  │  // 数据空间信息
│ 6. Export Section (0x06, 可选)  │  // 导出函数
│ 7. Element Section (0x07, 可选) │  // 引用函数索引
│ 8. Data Section (0x08, 可选)    │  // 数据初始值
│ 9. Code Section (0x09, 必选)    │  // 字节码指令
│ 10. Custom Sections (0x0A-0x0F) │  // 自定义段
└─────────────────────────────────┘
```

### 8.2 WebAssembly WASM 文件格式

WASM 文件结构：

```
WASM File Header (8 bytes):
┌─────────────────────────────────┐
│ magic: 0x0061736D               │  // "\0asm"
│ version: 0x00000001             │  // version 1
└─────────────────────────────────┘

Sections:
┌─────────────────────────────────┐
│ Type Section (1)                │  // 函数签名
│ Import Section (2)              │  // 导入
│ Function Section (3)            │  // 函数声明
│ Table Section (4)               │  // 表
│ Memory Section (5)              │  // 内存
│ Global Section (6)              │  // 全局变量
│ Export Section (7)              │  // 导出
│ Start Section (8)               │  // 启动函数
│ Element Section (9)             │  // 元素段
│ Code Section (10)               │  // 函数体
│ Data Section (11)               │  // 数据段
│ Data Count Section (12)         │  // 数据计数
└─────────────────────────────────┘
```

### 8.3 关键差异

| 特性 | COS3 SEF | WebAssembly WASM |
|------|----------|------------------|
| **文件标识** | `"sef"` (0x00736566) | `"\0asm"` (0x0061736D) |
| **段标识** | 0x01-0x0F | 0-12 |
| **应用信息** | ✅ App Section | ❌ 无 |
| **中间文件** | ✅ .wasm (符合WASM标准) | N/A |
| **链接文件** | ✅ .link | ❌ 无 |
| **字节序** | 小端 (i32, s16, s32, u16, u32) | 小端 (所有多字节) |
| **编码** | 固定长度 + LEB128 | LEB128 (可变长度) |

## 9. 执行模型差异

### 9.1 共同点

两者都是**基于栈的虚拟机** (Stack-based VM)：

```
栈式执行模型示例 (ADD指令):

执行前:  栈 [5, 3]  (栈顶在右)
执行:    pop 3, pop 5, add 5+3=8, push 8
执行后:  栈 [8]
```

### 9.2 差异点

| 特性 | COS3 | WebAssembly |
|------|------|-------------|
| **栈单元大小** | 4 字节 (固定) | 根据类型 (i32/i64/f32/f64) |
| **间接变量栈** | ✅ 16字节单元 (组合类型) | ❌ 无 |
| **局部变量** | 在执行器栈中 | 独立的局部变量数组 |
| **全局变量** | 在全局数据区 | 独立的全局变量数组 |
| **函数调用** | 创建栈帧 (保存BP/PC) | 激活记录 (activation record) |
| **返回值** | 压入调用者栈 | 放在返回值位置 |

## 13. 总结与建议

### 13.1 核心差异总结

| 维度 | COS3 | WebAssembly | GCOS VM实现建议 |
|------|------|-------------|----------------|
| **内存模型** | 分区+持久化 | 线性+易失 | ✅ 实现分区管理器 + Flash集成 |
| **事务管理** | 内置 | 无 | ✅ 实现事务管理器 |
| **应用生命周期** | 完整管理 | 宿主管理 | ✅ 实现应用管理器 + 多通道 |
| **安全管理** | 多层次 | 沙箱 | ✅ 实现运行域隔离 + 授权表 |
| **数据类型** | 仅整数 | 整数+浮点 | ✅ 仅实现整数运算 |
| **指令集** | 256+条 | 200+条 | ✅ 参考WASM实现COS3指令 |
| **文件格式** | SEF | WASM | ✅ 实现SEF解析器 |
| **资源需求** | 极低 | 无限制 | ✅ 优化代码大小和内存 |

### 13.2 可借鉴的WASM设计

虽然COS3和WASM差异很大，但以下WASM设计可以借鉴：

1. ✅ **栈式执行模型** - 两者相同，可直接参考
2. ✅ **指令编码** - LEB128可变长度编码可用于操作数
3. ✅ **模块化设计** - 模块导入/导出机制
4. ✅ **类型安全** - 加载时验证 + 运行时检查
5. ✅ **沙箱隔离** - 内存边界检查

### 13.3 COS3特有实现要点

GCOS VM必须实现的COS3特有功能：

1. ⭐⭐⭐ **分区内存管理**
   - 执行器栈 (256×4B)
   - 间接变量栈 (64×16B)
   - 全局数据区 (4KB)
   - 堆 (8KB, 非易失)
   - 模块程序区 (16KB)

2. ⭐⭐⭐ **事务管理**
   - 数据备份/恢复
   - 原子性保证
   - 嵌套事务

3. ⭐⭐⭐ **应用生命周期**
   - 安装/卸载
   - 选择/取消选择
   - 8通道管理
   - 生命周期状态机

4. ⭐⭐ **安全管理**
   - 运行域隔离
   - 接口授权表
   - 边界检查

5. ⭐⭐ **SEF文件解析**
   - 段解析
   - 符号链接
   - 模块实例化

6. ⭐ **Flash集成**
   - 使用eflash库
   - 持久化存储
   - 掉电恢复

### 13.4 实现优先级

根据差异分析，建议的实现优先级：

**P0 (最高优先级)**:
1. 分区内存管理器
2. 栈式执行引擎
3. 基础指令集 (控制流、算术、逻辑)

**P1 (高优先级)**:
4. SEF文件加载器
5. 应用管理器 (安装/选择/通道)
6. 事务管理器

**P2 (中优先级)**:
7. 安全管理器 (隔离/授权)
8. 高级指令集 (异常处理、复合指令)

**P3 (低优先级)**:
